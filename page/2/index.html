swift执行过程

发表于 2015-01-09   |  

刚学习ios 一直不明白从打开APP到现实View过程中程序究竟执行了什么，它是如何以最终的形式显示给我们的。

在swiftist摸爬打滚了一整子，自己总结了一点经验，可能不是很准确，但该文章会一直保持更新。

##屏幕（Screen）、窗口（Window）和视图（View）

  • UIScreen 代表一块物理屏幕；
  • UIWindow 代表一个窗口；
  • UIView 代表窗口里某一块矩形显示区域，用来展示用户界面和响应用户操作；
  • UILabel和UIButton，继承自UIView的特定UI控件，实现了特定的样式和行为。

##应用代理类（AppDelegate）与视图控制器（UIViewController）

在我们打开app的时候，程序会自动执行一个叫main的入口函数，调用UIApplicationMain函数来创建AppDelegate类的实例，并委托其实现app在不同生命周期的定制行为。

AppDelegate是应用程序代理类，其中定义了app进入不同生命周期（包括app启动动、闲置、进入后台、进入前台、激活、完全退出）时的回调方法。

AppDelegate内创建了一个UIWindow对象,UIWindow代表一个窗口，ios应用一般只有一个窗口，而os应用一般会有多个窗口。
我们可以建立一个UIViewController类来指定UIWindow.rootViewController，这样就可以在该视图控制器上绘制我们需要的视图控件

##视图控制器（UIViewController）生命周期

当我们的视图控制器与应用代理类绑定并在屏幕上显示时，代码的执行顺序

  1. alloc 创建对象，分配内存
  2. init(initWithNibName) 初始化对象
  3. loadView 从nib载入视图，通常这一不需要去干涉。除非你没有使用xib文件创建视图
  4. viewDidLoad 载入完成，可以进行自定义数据以及动态创建其他控件
  5. viewWillAppear 视图将出现在屏幕之前，马上这个视图就会被展现在屏幕上了
  6. viewDidAppear 视图已在屏幕上渲染完成，当一个视图被移除屏幕并且销毁的时候的执行顺序，这个顺序差不多和上面的相反
  7. viewWillDisappear 视图将被从屏幕上移除之前执行
  8. viewDidDisappear 视图已经被从屏幕上移除，用户看不到这个视图了
  9. viewDidUnload(不一定执行) 当发生内存警告（内存不足或其他原因）会销毁不是当前视图的其他视图以及附属子视图，开发者需要在该方法手动对viewLoad和viewDidLoad中创建的对象释放内存。防止在这个视图再次显示在屏幕上的时候，当调用viewLoad和viewDidLoad存在了多个视图对象。
  10. dealloc 视图被销毁，此处需要对你在init和viewDidLoad中创建的对象进行释放

代理模式

发表于 2015-01-07   |  

代理模式是在客户端与实体之间建立一个代理对象(proxy),客户端对实体进行操作全部委派给代理对象，隐藏实体的具体实现细节。

代理对象(proxy)还可以与业务代码分离，部署到另外的服务器。业务代码中通过RPC来委派任务。

####代理模式：

/*主题接口*/
interface IUser {
    public function getName() ;
}

/*真实类*/
class UserImpl implements IUser{    
    private $name = '' ;
    public __construct($name){
        $this->name = $name
    }

    public function getName(){
        return $this->name ;
    }
}

/*代理类*/    
class UserProxy Implements IUser{
    private $impl = null ;
    public __construct(IUser $userImpl){
        $this->impl = $userImpl ;
    }

    public function getName(){
        return $this->impl->getName() ;
    }
}

/*客户端*/
class Client{
    static function main(){
        $proxy  = new UserProxy(new UserImpl('Qmore')) ;
        echo $proxy->getName() ;
    }
}

Client::main()
// Qmore

代理模式与装饰模式、监视者模式有相似处，但代理是控制真实对象的访问，而后两者为对象添加额外的功能和触发额外的事件。

代理模式在很多情况下都非常有用,特别是你想强行控制一个对象的时候,比如:延迟加载,监视状态变更的方法等等

装饰器模式

发表于 2015-01-06   |  

当我们在一个类的功能上需要添加额外的功能时候，在传统的编程方式里要么过程式的在核心代码外包围一连串的额外功能代码或者写一个子类集成它重写这个类的方法。

使用装饰模式，我们可以动态地添加修改类的功能，仅需在运行时添加一个装饰器对象即可，实现了极大的灵活性。

####被装饰类：

class DataClass {
    function write(){
        echo "write some thing" ;
    }
}

####传统方式1——过程式：

class DataClass {

    function wirte(){

        /*包裹*/
        echo "before write some thing" ;

        echo "wirte some thing" ;

        /*结束*/
        echo "after write some thing" ;
    }

}

####传统方式2——继承父类：

class SubDataClass extends DataClass{
    function write(){
        /*包裹*/
        echo "before write some thing" ;

        parent::write() ;    

        /*结束*/
        echo "after write some thing" ;
    }
}

###装饰模式：

/*定义装饰器接口*/
interface Decorator{
    function before() ;
    function after() ;
}

/*定义装饰器*/
class SizeDecorator implments Decorator {
    function before(){
        echo '<div style="font-size:20px;">' ;
    }

    function after(){
        echo '<div>'
    }
}

/*主体*/
class DataClass {
    private $decorator = array() ;

    public function addDecorator(Decorator $de){
        $this->decorator[] = $de ;    
    }

    private function before(){
        foreach($this->decorator as $de){
            $de->before() ;
        }
    }

    private function after(){
        /*反转*/
        $des = array_reverse($this->decorator)
        foreach(des as $de){
            $de->after() ;
        }
    }

    function write(){
        $this->before() ;

        echo "wirte some thing" ;

        $this->after() ;
    }
}

/*控制器*/
$c = new DataClass() ;
$c->addDecorator(new SizeDecorator()) ;
$c->write() ;

观察者模式

发表于 2015-01-06   |  

当一个对象状态发生改变时，依赖它的对象全部收到通知，并执行相对应的裸机操作，这样的情况我们可以使用观察者模式。

场景：一个事件发生后，要执行一连串逻辑操作。传统的方式是在该事件发生后过程式的侵入处理逻辑代码。一旦这些代码越来越多，会便得难以维护。这种方式是耦合的，侵入不友好的，每次增加新的逻辑需要修改事件主体代码。

######观察者模式实现场景的解耦，非侵入式的通知与更新机制。

####传统方式：

class Event{
    function trigger(){
        echo "do something..." ;

        echo "逻辑1" ;

        echo "逻辑2" ;

        echo "逻辑3" ;

    }
}

$ev = new Event; 
$ev->trigger() ;

####观察者模式:

//观察者接口
interface Observer(){
    function update($args = null) ;
}

//观察者1号 2号 3号 等
class Ob1 implments Observer{
    function update($args = null){
        echo "逻辑1"
    }
}

//事件生成器
abstract EventGenerator {
    private    $observers = array() ;
    function addObserver(Observer $ob){
        $this->observers[] = $ob ;
    }
    function notify(){
        foreach($this->observers as $ob){
            $ob->update()
        }
    }    
}

//事件主体
class Event extends EventGenerator {
    function trigger(){
        echo "do something" ;
        //执行其他逻辑
        $this->notify() ;
    }
}

 /***控制器代码***/

 $event = new Event() ;
 //添加观察者
 $event->addObserver(new Ob1()) ;
 $event->addObserver(new Ob2()) ;
 $event->addObserver(new Ob3()) ;
 //执行主体
 $event->trigger() ;

php5.4新特新——Trait

发表于 2014-12-20   |  

PHP 5.4中的traits，是新引入的特性。其实际的目的，是为了有的场合想用多继承，但PHP又没多继承，于是就发明了这样的一个东西。
Traits可以理解为一组能被不同的类都能调用到的方法集合，但Traits不是类！不能被实例化。先来例子看下语法：

trait myTrait{
    function traitMethod1(){}
    function traitMethod2(){}
}

通过use myTraits，调用Traits中的方法了，调用这个traits的语法为：

class myClass{
    use myTrait;
}

$obj = new myClass();
$obj-> traitMethod1 ();
$obj-> traitMethod2 ();

Ajax跨域post问题

发表于 2014-12-10   |  

大多数跨域AJAX请求我们都使用JSONP ，参考 JSONP的简单原理

而jsonp是基于get请求的，当要传输的数据过大，只能基于POST请求，get形式无法满足我们的需求

可以使用CORS(跨域资源共享，Cross-Origin Resource Sharing)

CORS可以帮助我们解决AJAX跨域的问题，允许一个域的网络应用向另一个域提交跨域请求，要实现该功能十分简单，只需要由服务器返回一个相应header头，以php服务器为例子:

接口中包含以下头信息:

header(“Access-Control-Allow-Origin: *”)
//(*)表示允许任何域名向该服务器提交信息，也可以指定对应域名:
header(“Access-Control-Allow-Origin: http://www.fortest.com”)

jsonp的简单原理

发表于 2014-11-05   |  

jsonp 全称是 JSON with padding ,是为了解决跨域请求资源问题的产生的解决方案。

当我们需要在我们编写的代码内去请求其他域名的接口，但由于著名的同源策略，会因为安全问题而遭到拒绝。

$.ajax({
    url:’http://baidx.com/teach.php’,
    type:’get’
}) ;

页面会报一个这样的错误：XMLHttpRequest cannot load http://test.com/books.php. Origin http://localhost is not allowed by Access-Control-Allow-Origin.

###JSONP的原理:

虽然有同源策略的限制，但是并不是所有HTML资源都必须在同一个域名，我们经常会在HTML标签中嵌入的一些图片、js、iframe、css样式等都可以通过外链的形式加载，可以利用标签跨域请求来实现。JSONP正式利用Script标签实现跨域问题的。

###代码实现:

####服务器端:

<?php 
    $data = array(
    array( ‘id’=>123,’name’=>’煮酒论西游’),
    array(‘id’=>124,’name’=>’封神榜’) );
    $json = json_encode($data) ;

    $callbackName = $_GET[‘callback’] ;
    echo “$callbackName($json)” ;

####客户端:

<script>
    function showbooks(json){
        console.log(json)
    }
</script>
<script src = “http://baidx.com/teach.php?callback=showbooks”></script>

下面的script会返回 showbooks({xxxxxxx}) ,相当于回调了showbooks方法，实现了跨域请求问题。
